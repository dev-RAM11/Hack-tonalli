1. Vision General del Sistema

Tonalli es una plataforma educativa gamificada que ensena Web3 y blockchain a traves de cursos interactivos. Los usuarios aprenden conceptos como blockchain, Stellar, wallets, DeFi y NFTs, y al completar evaluaciones reciben:

  • XP (puntos de experiencia) — gamificacion interna
  • XLM (Lumens) — recompensas reales en criptomoneda via smart contract
  • Certificados verificables — credenciales W3C emitidas on-chain via ACTA sobre Stellar
  • NFTs conmemorativos — tokens no fungibles como prueba de logro
Blockchain en Tonalli: Los smart contracts se utilizan en dos areas especificas:
1. Certificaciones — ACTA Verifiable Credentials almacenadas en un vault on-chain
2. Rewards — Distribucion automatica de XLM con anti-doble-claim via contrato Soroban

6. Smart Contract: Learn-to-Earn Rewards

Ubicacion: contracts/learn-to-earn/src/lib.rs
Lenguaje: Rust (Soroban SDK v21)
Red: Stellar Testnet
Proposito: Distribuir recompensas XLM automaticamente a usuarios que completan cursos

Funcionalidad

Este contrato maneja un pool de XLM del cual se distribuyen recompensas a los usuarios que completan modulos y exámenes. Incluye proteccion anti-doble-claim y bonus por score perfecto.

Estructura de datos on-chain

pub struct RewardRecord {
    pub lesson_id: String,     // id_curso del modulo completado
    pub amount: i128,          // Cantidad en stroops (1 XLM = 10^7)
    pub timestamp: u64,        // Fecha de emision
}

Funciones del contrato

FuncionParametrosDescripcion
initialize admin, xlm_token Inicializa el contrato con la cuenta admin y la direccion del token XLM (SAC)
reward_user user, lesson_id, amount, score Envia XLM al usuario. Verifica anti-doble-claim. Si score=100 aplica bonus +10%
is_lesson_rewarded user, lesson_id Verifica si el usuario ya recibio reward por ese curso (anti-doble-claim)
get_user_total_rewards user Total de XLM ganados por el usuario
get_reward_history user Historial completo de rewards: [{lesson_id, amount, timestamp}]
deposit from, amount Fondea el pool de recompensas
withdraw to, amount Retiro de emergencia (solo admin)
pool_balance — Balance disponible en el pool

Flujo de reward

Usuario completa quiz con score >= 80% │ ▼ Backend: ChaptersService.submitQuiz() │ ├── Calcula XP ganado ├── Verifica anti-doble-claim: is_lesson_rewarded(user, id_curso) │ ▼ Smart Contract: reward_user(user, id_curso, amount, score) │ ├── Si score == 100 → amount * 1.10 (bonus +10%) ├── Transfiere XLM: pool → wallet del usuario ├── Registra RewardRecord on-chain └── Emite evento para auditoria │ ▼ Backend guarda txHash en base de datos

Seguridad

  • Anti-doble-claim: El contrato verifica is_lesson_rewarded(user, lesson_id) antes de pagar. Si el usuario ya recibio reward por ese id_curso, la transaccion se rechaza.
  • Solo admin: Unicamente la cuenta admin puede invocar reward_user, withdraw.
  • Pool finito: Los rewards solo se pagan si hay balance suficiente en el pool.

7. Smart Contract: Certificaciones ACTA

'Proveedor: ACTA (acta.build)
Contrato: CCOTIN3C6THFBEBYG6CJBOYEUPMPH5BHATWMCXPT52M6T7YZURJAC64O
Estandar: W3C Verifiable Credentials v2
Red: Stellar Testnet
Proposito: Emitir certificados verificables inmutables para graduados de cursos

Arquitectura de ACTA

ACTA utiliza un modelo de Vault (boveda) en el contrato inteligente de Stellar donde se almacenan las credenciales verificables. El flujo requiere:

  1. Vault: Almacen on-chain donde se guardan las credenciales del emisor (Tonalli)
  2. Wallet: Cuenta Stellar del admin para firmar transacciones y acceder al vault
  3. Issuer: Entidad autorizada para emitir credenciales dentro del vault

Estructura de la Credencial Verificable

{
  "owner": "GD7TV4RC...",                    // Wallet admin (vault owner)

Flujo de emision de certificado

Usuario aprueba Examen Final (score >= 80%) │ ▼ Frontend: ChapterQuiz → useIssueCertificate() │ ▼ Backend: POST /api/certificates/issue │ ├── CertificatesService.issueActaCertificate() │ │ │ ▼ │ ActaService.issueCredential(userId, id_curso, titulo, score) │ │ │ ├── 1. Genera vcId: "vc:tonalli:ch:{id_curso}:{timestamp}" │ ├── 2. Construye vcData (W3C VC standard) │ ├── 3. Llama ACTA API: POST /contracts/vc/issue (prepare) │ │,(obtiene XDR sin firmar) │ ├── 4. Firma XDR con STELLAR_ADMIN_SECRET │ ├── 5. Submit XDR firmado → txId en Stellar │ └── 6. Retorna { txId, vcId } │ ▼ Guarda en BD: ActaCertificate { actaVcId, txHash, examScore, status } │ ▼ Frontend muestra certificado con link a Stellar Explorer

Firma de transacciones (signXdr)

// acta.service.ts — Metodo de firma
private signXdr(unsignedXdr: string, networkPassphrase: string): string {
  const tx = TransactionBuilder.fromXDR(unsignedXdr, networkPassphrase);
  tx.sign(this.adminKeypair);
  return tx.toXDR();
}
Patron Prepare → Sign → Submit: ACTA usa un patron de dos fases. Primero se prepara la transaccion (se obtiene un XDR sin firmar), luego el backend la firma con la clave secreta del admin, y finalmente se envia la transaccion firmada. Esto mantiene las claves secretas siempre en el servidor.

8. Flujos Principales

8.1 Registro de usuario

1. Usuario → POST /auth/register (email, password, username, city) 2. Backend → Crea cuenta en MySQL con role='user', plan='free' 3. StellarService → Genera Keypair (publicKey + secretKey) 4. Friendbot → Fondea cuenta con 10,000 XLM (testnet) 5. Backend → Guarda stellarPublicKey, stellarSecretKey, isFunded=true 6. Retorna → JWT token + datos de usuario

8.2 Completar un curso

1. Usuario accede al curso (id_curso) 2. Completa Modulo 1: Lectura → Video → Quiz (5 preguntas) 3. Completa Modulo 2: Lectura → Video → Quiz (5 preguntas) 4. Completa Modulo 3: Lectura → Video → Quiz (5 preguntas) └── Progreso: 75% — Free users se detienen aqui 5. Modulo 4 (Examen Final): 20 preguntas, min 80% ├── Pro: paga $2 USD en XLM para acceder └── Max: acceso gratuito 6. Si aprueba: ├── +50 XP ├── Smart Contract: reward_user() → XLM a wallet ├── ACTA: issueCredential() → certificado on-chain └── Designer puede verificar completitud

8.3 Rol Designer: verificar completitud

1. Designer accede al panel de cursos 2. Selecciona un curso (id_curso) 3. Ve el object array de usuarios inscritos: [ { userId, username, plan, modulosCompletados, porcentaje, examenAprobado, certificadoEmitido } ] 4. Filtra por usuarios que completaron al 100% 5. Verifica que el certificado ACTA fue emitido on-chain 6. Puede consultar status: GET /certificates/verify?vcId=... → Retorna { valid: true, onChainStatus: "valid" }

Inicializacion automatica ACTA

Al arrancar el backend con ACTA_API_KEY y STELLAR_ADMIN_SECRET configurados, el sistema automaticamente:

  1. Conecta con ACTA API testnet
  2. Obtiene actaContractId y networkPassphrase
  3. Crea el vault para el admin (si no existe)
  4. Autoriza al admin como issuer (si no esta autorizado)
  5. Queda listo para emitir credenciales
